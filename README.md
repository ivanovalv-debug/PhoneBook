Архитектурный шаблон MVVM
Цель работы: Изучить архитектурный шаблон Model-View-ViewModel (MVVM) и закрепить практические навыки его применения при разработке настольных приложений на 
платформе Windows Presentation Foundation.
Задание:
Разработать приложение «Телефонная книга» на платформе WPF с использованием паттерна MVVM. Приложение должно обеспечивать управление списком контактов 
(добавление, удаление, просмотр) и соответствовать следующим требованиям:
Реализовать модель Contact с полями Name (имя контакта) и Phone (номер телефона).
Создать ViewModel (MainViewModel) с командами добавления и удаления контактов
(AddCommand, DeleteCommand) Команда AddCommand – без параметра, DeleteCommand – с параметром.
Использовать ObservableCollection для хранения списка контактов с автоматическим обновлением UI.
Реализовать валидацию данных: имя контакта не должно быть пустым, номер телефона должен соответствовать формату (например, +7XXXXXXXXXX или без кода страны).
Отображать список контактов в элементе DataGrid или ListView с возможностью выбора контакта для удаления.
Обеспечить привязку данных (Data Binding) между XAML-разметкой и свойствами ViewModel.
Требования к коду: исходный код должен содержать комментарии, поясняющие назначение каждого компонента в контексте паттерна MVVM. Пользовательский интерфейс 
должен быть простым и функциональным, без излишнего оформления.
Теоретическое обоснование используемых паттернов
Реализация приложения «Телефонная книга» базируется на интеграции структурного паттерна проектирования MVVM (Model-View-ViewModel), который разделяет приложение на три независимых слоя. Данный подход решает проблему жесткой связности между пользовательским интерфейсом и бизнес-логикой.
Паттерн MVVM:
Назначение: Обеспечивает четкое разделение ответственности между данными (Model), визуальным представлением (View) и логикой отображения/управления (ViewModel).
Обоснование использования:
Слабая связанность (Loose Coupling): View не знает о существовании Model, а ViewModel не имеет прямых ссылок на элементы UI (кнопки, текстовые поля). Это позволяет изменять дизайн приложения без риска сломать бизнес-логику.
Тестируемость: Поскольку ViewModel не зависит от WPF-библиотек, её логику (валидацию, добавление контактов) можно покрывать модульными тестами без запуска графического интерфейса.
Data Binding: Использование механизма привязки данных WPF устраняет необходимость писать код в code-behind для синхронизации значений полей ввода со свойствами объектов.
Ключевые механизмы реализации:
INotifyPropertyChanged: Интерфейс, позволяющий объектам уведомлять View об изменении своих свойств. Без него интерфейс не обновлялся бы автоматически при изменении данных в коде.
ICommand: Интерфейс для инкапсуляции действий пользователя (нажатие кнопок). Позволяет привязывать логику выполнения команд непосредственно в XAML, избегая обработчиков событий (Click).
ObservableCollection<T>: Специализированная коллекция, которая отправляет уведомления при добавлении или удалении элементов. Это критически важно для динамических списков (DataGrid), чтобы UI мгновенно реагировал на изменения в наборе данных.
Теоретическая оценка ожидаемого результата
В результате применения паттерна MVVM программная система должна демонстрировать следующие характеристики:
Модульность и сопровождаемость: Код разделен на логические блоки. Изменение верстки (XAML) не требует перекомпиляции логики (C#), и наоборот.
Автоматическая синхронизация UI: При вводе данных в текстовые поля или добавлении контакта в список, интерфейс обновляется мгновенно благодаря механизму Data Binding и событиям PropertyChanged.
Централизация логики управления: Все действия пользователя (добавление, удаление) обрабатываются через команды в ViewModel, что делает поток выполнения программы предсказуемым и линейным.
Надежность ввода данных: Валидация данных происходит на уровне Model/ViewModel до того, как данные будут сохранены в коллекцию, что предотвращает появление некорректных записей в интерфейсе.
Описание выполненных действий
Подготовка инфраструктуры и базовых абстракций
Создание структуры проекта: Проект организован в папки Models, ViewModels и Views для обеспечения физической модульности кода.
Реализация базового класса ObservableObject: Создан абстрактный класс, реализующий интерфейс INotifyPropertyChanged. Он содержит метод OnPropertyChanged и вспомогательный метод Set<T>, который упрощает установку значений свойств и автоматический вызов уведомлений об изменении.
Реализация классов команд RelayCommand: Созданы две версии класса, реализующего интерфейс ICommand (с параметром и без). Они делегируют выполнение действий методам ViewModel и управляют доступностью кнопок через предикат CanExecute.
Реализация слоя Model (Данные)
Создание класса Contact: Модель представляет собой простой объект данных с полями Name (строка) и Phone (строка).
Валидация данных: Реализован статический метод Validate, который проверяет, что имя не пустое, а номер телефона соответствует формату (10 или 11 цифр, возможно с префиксом +7 или 8). Конструктор класса выбрасывает исключение при передаче некорректных данных, обеспечивая целостность модели.
Реализация слоя ViewModel (Логика представления)
Создание MainViewModel: Центральный класс, наследующий ObservableObject.
Управление состоянием:
Свойства Name и Phone связаны с полями ввода.
Свойство SelectedContact хранит ссылку на элемент, выбранный пользователем в таблице.
Коллекция Contacts типа ObservableCollection<Contact> хранит список всех контактов.
Реализация команд:
AddCommand: Проверяет валидность введенных данных через Contact.Validate. Если данные корректны, создает новый объект Contact и добавляет его в коллекцию Contacts. После этого очищает поля ввода.
DeleteCommand: Принимает выбранный контакт как параметр. Удаляет объект из коллекции Contacts, если он не равен null.
Реализация слоя View (Пользовательский интерфейс)
Разметка MainWindow.xaml:
Использован контейнер Grid для расположения элементов.
Элементы TextBox привязаны к свойствам Name и Phone через {Binding ...} с режимом обновления UpdateSourceTrigger=PropertyChanged (для мгновенной реакции).
Кнопки Button привязаны к командам AddCommand и DeleteCommand через свойство Command. Для кнопки удаления дополнительно указан CommandParameter="{Binding SelectedContact}".
Элемент DataGrid привязан к коллекции Contacts через ItemsSource и к выбранному элементу через SelectedItem.
Настройка DataContext: В конструкторе окна MainWindow.xaml.cs свойству DataContext присвоен экземпляр MainViewModel, что активирует механизм привязки данных для всего окна.
Результат выполненной работы
Тестирование достигнутого поведения
В ходе выполнения программы были успешно проверены ключевые сценарии взаимодействия пользователя с системой:
Добавление валидного контакта (Scenario 1):
Пользователь вводит имя "Иван" и телефон "+79991234567".
При нажатии кнопки «Добавить» создается объект Contact.
Запись мгновенно появляется в нижней таблице (DataGrid).
Поля ввода автоматически очищаются, становясь готовыми к новому вводу.
Обработка некорректных данных (Scenario 2):
Попытка добавить контакт с пустым именем или неверным форматом телефона (например, "abc").
Кнопка «Добавить» остается неактивной (или действие игнорируется, в зависимости от реализации CanExecute), либо данные не проходят валидацию в конструкторе модели.
Список контактов не изменяется, целостность данных сохранена.
Удаление контакта (Scenario 3):
Пользователь выделяет строку в таблице.
Кнопка «Удалить» становится активной.
При нажатии выбранный объект удаляется из ObservableCollection.
Интерфейс автоматически перестраивается, удаляя соответствующую строку из таблицы.
Динамическое обновление UI (Scenario 4):
Проверено, что изменение текста в поле ввода немедленно отражается в свойстве ViewModel (благодаря UpdateSourceTrigger=PropertyChanged).
Добавление нового элемента в коллекцию сразу же отображается в DataGrid без необходимости вручную обновлять таблицу.
Сравнение с теоретической оценкой
Полученные практические результаты полностью подтверждают теоретические принципы паттерна MVVM:
Отсутствие Code-Behind логики: Файл MainWindow.xaml.cs содержит только вызов InitializeComponent() и установку DataContext. Вся логика обработки нажатий и валидации вынесена в ViewModel.
Автоматическая синхронизация: Использование ObservableCollection и INotifyPropertyChanged обеспечило двустороннюю связь данных. Изменения в коде мгновенно видны пользователю, и ввод пользователя мгновенно доступен в коде.
Изоляция слоев: Модель Contact не знает о существовании окон или кнопок. ViewModel оперирует только данными и командами. View занимается исключительно отображением. Это подтверждает эффективность разделения ответственности.
Вывод
В ходе лабораторной работы было разработано приложение «Телефонная книга» на платформе WPF с использованием архитектурного паттерна MVVM. Практическая реализация подтвердила, что использование INotifyPropertyChanged, ICommand и ObservableCollection<T> позволяет создать гибкую, легко тестируемую и поддерживаемую архитектуру. Разделение кода на Model, View и ViewModel исключает смешивание бизнес-логики и интерфейса, что является стандартом современной разработки десктопных приложений на .NET.
