Навигация в MVVM-приложениях. Подход ViewModel-First
Цель работы: изучить механизмы навигации между экранами в MVVM-приложении без использования стандартных средств WPF (Frame/Page).

Задание:
Выполнить рефакторинг приложения «Телефонная книга» в архитектуру Shell (Оболочка).
Требования к доработке:

•	Создать структуру папок: Разделить Views и ViewModels по папкам (Views и ViewModels).
•	Рефакторинг: переименовать ViewModel в ContactsListViewModel, а MainWindow.xaml — в ContactsListView (UserControl).
•	Создать Shell (Главное окно) с меню и областью контента (ContentControl) и MainWindowViewModel.
•	Разработать интерфейс INavigationService с методом NavigateTo<TViewModel>() и реализовать сервис навигации.
•	Настроить DI: зарегистрировать ViewModels, определить DataTemplate в App.xaml. Создать дополнительный экран «О программе» (AboutViewModel / AboutView).
Требования к коду: исходный код должен содержать комментарии, поясняющие назначение зарегистрированных сервисов и время их жизни (Lifetime).
Теоретическое обоснование используемых паттернов
Рефакторинг приложения «Телефонная книга» выполнен с применением архитектуры Shell (Оболочка) и паттерна ViewModel-First навигации.
Паттерн Shell (Оболочка):
•	Назначение: Главное окно приложения выступает в роли универсального контейнера, содержащего меню навигации и динамическую область контента (ContentControl).
•	Обоснование: Данный подход заменяет создание множества окон или использование навигационных фреймов. Приложение сохраняет единый визуальный стиль и структуру управления, подгружая разные экраны в одну область. Это упрощает управление состоянием и позволяет реализовать единую точку входа для пользователя.
ViewModel-First навигация:
•	Назначение: При переключении экранов в первую очередь создается экземпляр ViewModel (через контейнер зависимостей), а WPF автоматически подбирает соответствующее визуальное представление (View) на основе настроек DataTemplate.
•	Обоснование: Этот метод обеспечивает полную поддержку Dependency Injection. ViewModel не имеет прямых ссылок на View или другие компоненты интерфейса, что гарантирует слабую связность. Кроме того, это позволяет тестировать логику навигации без запуска графического интерфейса.
Сервис навигации (INavigationService):
•	Назначение: Централизация логики переключения между экранами.
•	Обоснование: Вместо прямого изменения свойств окна, все экраны используют единый сервис для перехода. Это позволяет легко масштабировать приложение, добавлять новые экраны и передавать параметры между ними (например, выбранный контакт для редактирования) через интерфейс INavigationAware.

Теоретическая оценка ожидаемого результата
После рефакторинга система должна демонстрировать следующие характеристики:
•	Модульность: Каждый экран (список контактов, редактирование, о программе) представляет собой независимый модуль (UserControl + ViewModel), который можно разрабатывать отдельно.
•	Динамическая подгрузка: Область контента (ContentControl) автоматически заменяет один экран на другой при изменении активной ViewModel без перезагрузки окна.
•	Слабая связность: ViewModel экранов не знают о существовании конкретных визуальных элементов. Вся коммуникация происходит через абстрактный сервис навигации.
•	Расширяемость: Добавление нового функционала требует только создания новой пары View/ViewModel и регистрации связи в ресурсах приложения.

Описание выполненных действий
1. Подготовка структуры проекта
   •	В проекте созданы папки Views и ViewModels для упорядочивания файлов по слоям архитектуры.
   •	Существующее главное окно переименовано в ContactsListView и преобразовано из типа Window в UserControl.
   •	Основной класс ViewModel переименован в ContactsListViewModel и перемещен в соответствующую папку.
3. Реализация сервисов навигации
   •	Разработан интерфейс INavigationService, описывающий методы для перехода между экранами и свойство для хранения текущей активной ViewModel.
   •	Создан интерфейс INavigationAware, позволяющий передавать параметры (например, объект контакта) при переходе на новый экран.
   •	Реализован класс NavigationService, который получает ViewModel из DI-контейнера, вызывает метод приема параметров (если интерфейс поддерживается) и обновляет свойство текущего экрана, уведомляя интерфейс об изменениях.
5. Создание новых экранов
   •	Экран редактирования: Созданы ContactEditView и ContactEditViewModel. ViewModel реализует INavigationAware для получения данных выбранного контакта. Реализованы команды сохранения и отмены с возвратом к списку.
   •	Экран «О программе»: Созданы простые AboutView и AboutViewModel для демонстрации переключения между независимыми экранами.
7. Создание оболочки приложения (Shell)
   •	Создано новое главное окно MainWindow, содержащее верхнее меню с кнопками навигации и область контента (ContentControl).
   •	Реализована MainWindowViewModel, управляющая командами кнопок меню и инициирующая загрузку стартового экрана при запуске.
9. Настройка связей (DataTemplate) и DI
    •	В ресурсах приложения (App.xaml) настроены шаблоны данных (DataTemplate), связывающие типы ViewModel с соответствующими визуальными элементами (Views). Это позволяет WPF автоматически «рисовать» нужный UserControl при появлении соответствующей ViewModel.
   •	В App.xaml.cs настроен контейнер зависимостей:
   o	Сервисы навигации и главное окно зарегистрированы как Singleton (существуют в единственном экземпляре).
   o	ViewModel экранов зарегистрированы как Transient (создаются заново при каждом переходе для обеспечения чистого состояния).
   •	Удалены устаревшие способы запуска приложения (StartupUri и привязка DataContext в XAML), запуск перенесен в код App.xaml.cs.

Вывод
В ходе лабораторной работы №11 архитектура приложения «Телефонная книга» была полностью модернизирована. Переход от монолитного окна к архитектуре Shell с использованием подхода ViewModel-First позволил решить проблему жесткой связности и подготовки приложения к масштабированию.
Реализация навигации через ContentControl и DataTemplate в связке с сервисом INavigationService обеспечивает гибкое управление экранами, полную поддержку Dependency Injection и возможность передачи данных между модулями. Приложение стало модульным, легко тестируемым и готовым к внедрению сложных сценариев взаимодействия с пользователем.
Структура проекта соответствует стандартам MVVM:
•	Главное окно (Shell) управляет навигацией.
•	Экраны (Views) отвечают только за отображение.
•	Логика (ViewModels) инкапсулирована и управляется контейнером зависимостей.
