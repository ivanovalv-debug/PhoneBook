Dependency Injection и паттерн «Сервис» в MVVM-приложениях
Цель работы: изучить принцип внедрения зависимостей (Dependency Injection, DI) и паттерн «Сервис» для решения проблемы жёсткой связности компонентов в MVVM-приложениях.

Задание:
Выполнить рефакторинг приложения «Телефонная книга» из Лабораторной работы №9 с применением паттернов Dependency Injection и Service.
Требования к доработке:
Подключить NuGet-пакет Microsoft.Extensions.DependencyInjection.
Разработать интерфейс IDialogService и его реализацию DialogService, поддерживающую вызов информационных,
предупреждающих, ошибочных сообщений и запросов подтверждения (Да/Нет).
Модифицировать класс ViewModel: внедрить IDialogService через конструктор. Использовать сервис в командах:
 - При удалении контакта запрашивать подтверждение (ShowConfirmation). Если пользователь отказывается — не удалять
контакт.
 - При добавлении контакта проверять наличие дубликатов по номеру телефона. Если номер есть — предупреждать (ShowWarning) и отменять добавление.
 - При успешном добавлении выводить информационное сообщение (ShowInfo).
Изменить точку входа приложения: убрать атрибут StartupUri из App.xaml. В App.xaml.cs настроить IoC-контейнер, зарегистрировать сервисы, ViewModel и главное окно. Инициализировать и запустить приложение вручную.
Убрать жёсткую привязку DataContext из XAML-разметки MainWindow.xaml.
Требования к коду: исходный код должен содержать комментарии, поясняющие назначение зарегистрированных сервисов и время их жизни (Lifetime).
Теоретическое обоснование используемых паттернов
Рефакторинг приложения «Телефонная книга» выполнен с применением двух ключевых архитектурных принципов:
Паттерн «Сервис» (Service Pattern):
Назначение: Абстрагирует взаимодействие с внешними зависимостями (диалоговые окна, файловая система, API) через интерфейсы.
Обоснование: Вместо прямого вызова MessageBox.Show() из ViewModel, создаётся интерфейс IDialogService, который описывает что нужно сделать, но не как. Это устраняет жёсткую связность ViewModel с WPF-библиотеками (PresentationFramework.dll) и делает код тестируемым.
Dependency Injection (внедрение зависимостей):
Назначение: Объекты не создают свои зависимости через new, а получают их извне (обычно через конструктор).
Обоснование: Использован Constructor Injection как наиболее надёжный способ, гарантирующий полную инициализацию объекта. ViewModel объявляет зависимость от IDialogService в конструкторе, а IoC-контейнер автоматически предоставляет реализацию.
IoC-контейнер (Inversion of Control):
Назначение: Автоматизирует создание объектов и управление их жизненным циклом.
Обоснование: Использован Microsoft.Extensions.DependencyInjection для регистрации сервисов с указанием lifetime (Singleton для DialogService, Transient для ViewModel). Контейнер разрешает граф зависимостей и подставляет их в конструкторы.
Теоретическая оценка ожидаемого результата
После рефакторинга система должна демонстрировать:
Слабую связность: ViewModel зависит только от абстракции IDialogService, а не от конкретной реализации.
Тестируемость: ViewModel можно покрыть юнит-тестами, подменив DialogService на MockDialogService, который не показывает окна.
Контроль над жизненным циклом: IoC-контейнер управляет созданием объектов, позволяя легко масштабировать приложение (добавлять навигацию, новые сервисы).
Чистую архитектуру: Удаление StartupUri и Window.DataContext из XAML передаёт ответственность за создание объектов в код (App.xaml.cs).
Описание выполненных действий
1. Подготовка инфраструктуры
Установлен NuGet-пакет Microsoft.Extensions.DependencyInjection.
Создана папка Services для размещения сервисных классов.
2. Реализация паттерна «Сервис»
Интерфейс IDialogService: Описаны методы ShowInfo, ShowWarning, ShowError, ShowConfirmation.
Класс DialogService: Реализован с использованием System.Windows.MessageBox. Зарегистрирован как Singleton (не хранит состояние).
3. Модификация ViewModel
Добавлена зависимость IDialogService через конструктор (Constructor Injection).
Команда AddCommand:
Проверка на дубликаты по номеру телефона через Contacts.Any().
При дубликате → _dialogService.ShowWarning() и отмена добавления.
При успехе → _dialogService.ShowInfo().
Команда DeleteCommand:
Запрос подтверждения через _dialogService.ShowConfirmation().
Если пользователь нажал «Нет» → удаление отменяется.
4. Настройка IoC-контейнера
App.xaml: Удалён атрибут StartupUri="MainWindow.xaml".
App.xaml.cs:
Создан ServiceCollection.
Зарегистрированы сервисы:
AddSingleton<IDialogService, DialogService>()
AddTransient<MainViewModel>() (новый экземпляр для будущей навигации)
AddSingleton<MainWindow>(sp => { ... }) с явной установкой DataContext.
Вызван BuildServiceProvider() для создания контейнера.
Получено окно через GetRequiredService<MainWindow>() и вызван Show().
5. Очистка View
MainWindow.xaml: Удалён блок <Window.DataContext>.
MainWindow.xaml.cs: Удалена строка DataContext = new MainViewModel() (теперь DataContext устанавливается из контейнера).
Результат выполненной работы
Тестирование достигнутого поведения
Проверены три ключевых сценария:
Успешное добавление (Scenario 1):
Введены уникальные имя и телефон.
Появилось информационное окно «Контакт успешно добавлен!».
Контакт отображается в таблице.
Обнаружение дубликата (Scenario 2):
Введён существующий номер телефона.
Появилось предупреждение ShowWarning «Контакт с таким номером уже существует!».
Контакт не добавлен в коллекцию.
Подтверждение удаления (Scenario 3):
Выбран контакт, нажата кнопка «Удалить».
Появился запрос подтверждения ShowConfirmation.
При нажатии «Да» → контакт удалён, показано ShowInfo.
При нажатии «Нет» → контакт остался в списке.
Сравнение с теоретической оценкой
Полученные результаты подтверждают принципы DI и Service Pattern:
Слабая связность: ViewModel не содержит ссылок на System.Windows.MessageBox. Взаимодействие происходит через IDialogService.
Тестируемость: Логика команд (AddContact, DeleteContact) может быть протестирована без запуска UI (достаточно передать mock-реализацию сервиса).
Контроль над lifetime: Контейнер корректно создаёт один экземпляр DialogService на всё приложение и новые экземпляры ViewModel при необходимости.
Чистая архитектура: Удаление StartupUri и DataContext из XAML позволило централизовать создание объектов в App.xaml.cs, что упрощает масштабирование (добавление навигации, логгирования, работы с API).
